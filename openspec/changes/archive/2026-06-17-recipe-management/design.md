## Context

`bootstrap-auth` 已经归档，后端具备 NestJS、Drizzle、PostgreSQL、JWT、`chef`/`diner` 角色守卫和共享 Zod 校验。当前还没有菜谱数据模型，后续 PC 后台、移动端浏览、点餐加菜都会依赖同一套菜谱数据。本 change 只做后端 API 与数据库，先建立稳定的数据边界。

现有项目没有全局 `/api` 前缀，认证接口使用 `auth/*`，本次接口沿用同一风格，使用 `categories/*` 与 `dishes/*`。

## Goals / Non-Goals

**Goals:**
- 新增分类、菜谱、食材、做法步骤四张表，与项目冻结数据模型保持一致。
- 提供分类管理 API，支持排序。
- 提供菜谱管理 API，支持分类、封面路径、描述、B 站视频、难度、启用状态、食材、做法步骤。
- 提供登录用户可用的浏览 API：分类列表、菜谱列表、菜谱详情。
- 管理写操作仅允许 `chef`；`diner` 只能看到启用菜谱。
- 将请求/响应 schema 与类型放入 `packages/shared`，后端校验、未来前端复用。
- 覆盖核心服务与 API 测试。

**Non-Goals:**
- 不实现图片上传；`cover_image` 与 step `image` 只保存本地路径字符串。
- 不实现 PC 管理后台页面。
- 不实现移动端菜谱页面。
- 不实现点餐、订单、分享访客免登录。
- 不实现复杂全文搜索、标签、收藏、评分、营养信息。

## Decisions

### 决策 1：菜谱详情用聚合写入与聚合读取
- **选择**：`POST /dishes` 和 `PATCH /dishes/:id` 接受菜谱基础字段、`ingredients[]`、`steps[]`；详情接口一次返回基础信息、分类、食材和按 `step_no` 排序的做法步骤。
- **理由**：主厨录入菜谱时天然是一个整体，后台表单也会一次提交；移动端查看详情也需要一次拿完整内容。
- **替代方案**：食材和步骤做独立 CRUD。被暂缓，因为首版会增加很多碎接口，前端交互收益不明显。

### 决策 2：更新菜谱时整体替换食材与步骤
- **选择**：更新菜谱详情时，在事务内更新 `dishes`，删除旧 `ingredients` / `recipe_steps`，再插入新的列表。
- **理由**：步骤重排、食材改量在 UI 中通常是列表整体编辑；整体替换逻辑更简单，避免每个子项维护单独 patch/delete/reorder API。
- **取舍**：子项 id 稳定性较弱，但当前没有外部引用食材或步骤，影响可接受。

### 决策 3：菜谱删除采用停用语义
- **选择**：菜谱通过 `is_active` 控制上下架；`DELETE /dishes/:id` 等价于停用，另提供 `PATCH /dishes/:id/active` 显式启用/停用。
- **理由**：未来订单会引用菜谱，硬删除容易破坏历史数据。停用后食客不可见，主厨仍可维护和恢复。
- **替代方案**：硬删除菜谱。被否决，后续接点餐时迁移成本高。

### 决策 4：分类允许未使用时删除
- **选择**：`DELETE /categories/:id` 仅在没有菜谱引用时允许；已被引用返回 409。
- **理由**：分类属于组织信息，未使用时删除安全；被菜谱引用时删除会造成列表和筛选语义不清。

### 决策 5：读取权限按角色控制可见范围
- **选择**：所有菜谱接口都要求登录。`diner` 列表与详情只返回 `is_active=true` 的菜谱；`chef` 可通过 query 查看全部或指定启用状态。
- **理由**：当前产品是主厨和食客登录使用，访客免登录分享属于未来能力。先用登录态保护接口，同时保留 chef 管理视角。

### 决策 6：难度用共享枚举约束
- **选择**：`difficulty` 使用 `easy | medium | hard`，数据库以 `varchar` 存储，shared Zod schema 约束。
- **理由**：足够表达家常菜复杂度，也比自由文本更利于筛选和前端展示。

## Risks / Trade-offs

- **[整体替换子项导致并发覆盖]** → 首版是单主厨低并发场景，风险低；后续如需多人编辑再加版本号或细粒度子项 API。
- **[图片只存路径，缺少上传闭环]** → 本 change 只定义存储字段和 API 校验；上传与 nginx 托管可在独立 change 中实现。
- **[B 站字段格式可能不统一]** → 先允许 BV 号或链接字符串，后端做长度校验；未来前端展示时再抽取播放器适配逻辑。
- **[分类删除 409 需要前端处理]** → 这是保护数据一致性的必要反馈；PC 后台后续展示“该分类下仍有菜谱，不能删除”。

## Migration Plan

- 新增 Drizzle schema：`categories`、`dishes`、`ingredients`、`recipe_steps`。
- 生成并提交 migration；本地执行 migration 后不会影响现有 `users` 数据。
- 回滚时可删除新增表；当前没有生产数据，回滚风险低。
