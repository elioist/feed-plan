## Context

当前点菜闭环后端接口与测试已经存在，`POST /meals/:id/orders` 支持登录用户加菜，重复加菜会累加数量。实测移动端点不了单时，失败原因不可见；同时 Expo 端默认 `http://localhost:3000` 在真机或部分模拟器中通常指向设备自身，不是开发机后端。

用户管理已经有基础列表、新建用户、修改角色和删除用户，但角色仍是单值且固定在 `chef` / `diner`，角色页展示来自 shared 常量，权限点不可新增、不可搜索、不可授权配置。用户明确要求角色和角色权限点都必须可配置，且支持多角色，不接受写死方案。

## Goals / Non-Goals

**Goals:**

- 让 mobile 购物车和菜单详情两条点单路径都能给出明确成功/失败反馈。
- 补充 Expo API base 配置说明，降低真机/模拟器连接后端的踩坑概率。
- 增加当前用户修改密码和有权限用户重置他人密码能力。
- 将用户授权改为多角色模型，用户可以绑定多个角色。
- 将角色和权限点改为数据库配置，支持搜索、新增、编辑、删除和授权。
- 后端敏感接口按权限点校验，不再按固定角色名校验。
- 移除 shared、admin、mobile 中固定 `chef` / `diner` 角色配置和固定权限矩阵。

**Non-Goals:**

- 不实现组织、多租户或行级数据权限。
- 不实现访客分享链路权限模型；未来分享功能仍单独设计。
- 不实现 token 强制失效、全端下线或审计日志；本次只保证密码哈希更新和响应不泄漏敏感字段。
- 不将前端直接依赖 `packages/db`；所有授权数据仍通过 server API 和 `packages/api-client` 获取。

## Decisions

### 1. 采用数据库驱动的多角色 RBAC

新增 `roles`、`permissions`、`user_roles`、`role_permissions`。用户与角色是多对多，角色与权限点是多对多。`users` 表不再以固定 `role` 字段作为授权依据。

角色字段包含 `key`、`name`、`description`、`is_system`、`created_at`、`updated_at`。权限点字段包含 `key`、`name`、`description`、`is_system`、`created_at`、`updated_at`。`key` 用于稳定引用和 API 展示，`name` / `description` 用于后台配置展示。

`is_system` 只表示系统初始化数据，防止误删最后管理入口或内置保护数据；它不代表角色/权限点不可配置。系统角色和权限点仍允许改名、改描述和调整授权，删除时需要遵守安全约束。

### 2. 权限点可配置，接口动作保持系统动作标识

后端接口必须有稳定的受保护动作标识，例如用户管理、角色管理、菜谱管理、点餐结单等。动作标识属于代码级系统边界，不等同于权限点。

权限点是数据库记录。角色拥有权限点；权限点可以映射到一个或多个受保护动作。Guard 在请求时读取用户拥有的权限点，再检查这些权限点是否覆盖当前动作。这样管理员可以新增权限点、改名、调整角色授权，也可以在后续扩展动作映射，而不会把权限矩阵写死在 shared 或页面里。

为了控制本次范围，第一版可先由 seed 创建内置动作映射；后台先提供角色-权限授权和权限点 CRUD。动作映射表保留数据库结构，后续如果需要可再做可视化配置页。敏感接口若没有映射到任何权限点，默认拒绝访问，避免“忘记配置就放行”。

### 3. 登录态返回角色和权限摘要

JWT 负载从单一 `role` 改为包含 `roles` 与 `permissions` 摘要。`GET /auth/me` 返回当前用户 id、username、roles、permissions。前端展示角色名时使用服务端返回的角色摘要，不再用 shared 常量推导。

权限变更后，用户重新登录或刷新当前用户信息即可拿到新的权限摘要。本次不实现实时 token 撤销；后台权限操作成功后刷新列表即可。

### 4. 密码接口放在 auth/users 边界内

当前用户改密码属于认证会话能力，使用 `PATCH /auth/password`，要求提交旧密码和新密码。有用户管理权限的用户重置他人密码属于用户管理能力，使用 `PATCH /users/:id/password`，禁止通过该接口重置自己的密码。

密码仍使用 bcrypt 哈希；响应体不返回密码或 hash。改密码成功后，当前用户可继续保持登录，后续再考虑 token 轮换或全端下线。

### 5. Mobile 点单修复优先解决配置与可观测性

保留现有 `packages/api-client` 请求封装，不在 mobile 内新建 API 层。mobile 捕获 `ApiError`，按 401、网络错误、409、404、400 给出不同文案，并在开发环境中包含 API base 提示。

购物车提交多道菜仍按当前顺序调用 `addOrder`。如果中途失败，提示具体失败原因，并刷新相关 query，避免 UI 停在“看起来成功但服务端失败”的状态。

## Risks / Trade-offs

- [Risk] 动态权限系统可能导致管理员误删全部管理权限。→ 禁止用户移除自己最后一个有效管理权限；删除角色/权限点前校验是否会造成无管理入口。
- [Risk] JWT 中权限摘要可能变旧。→ 权限修改后通过刷新当前用户信息获取最新摘要；强制 token 失效留给后续。
- [Risk] 权限动作映射如果漏配会影响接口可用性。→ 受保护动作无映射时默认拒绝，并在 seed/test 中覆盖内置管理动作。
- [Risk] 购物车多请求提交可能部分成功。→ 本次先提示失败并刷新菜单状态；批量下单事务接口留给后续优化。
