## Context

当前 users 表只有 `id`、`username`、`password_hash`、`created_at` 四个字段。管理后台用户管理页支持新建、分配角色、重置密码和删除，但无法修改用户基本信息。头像上传功能已存在于菜谱模块（`dishes.uploadImage`），可复用上传接口和存储路径。

## Goals / Non-Goals

**Goals:**
- 支持主厨在管理端编辑任意用户的基本信息（用户名、头像）。
- 支持用户在移动端修改自己的头像。
- 用户名修改需校验唯一性，头像支持 1:1 裁剪后上传。
- 复用现有图片上传能力和存储目录。

**Non-Goals:**
- 不支持用户自行修改密码（已有独立流程）。
- 不支持修改用户角色（已有独立流程）。
- 不引入图片 CDN 或云存储，继续使用服务器本地目录 + nginx 托管。

## Decisions

### 1. 头像存储复用菜谱图片上传接口

现有 `POST /uploads/image` 接口将图片保存到 `uploads/` 目录并返回相对路径。头像直接复用此接口，不新建独立的上传端点。

备选方案是为头像创建独立上传接口。但它与菜谱图片上传逻辑完全一致，拆分只会增加维护成本。

### 2. 用户名修改走 PATCH 接口，带唯一性校验

`PATCH /users/:id` 接口接收可选的 `username` 和 `avatar` 字段。修改用户名时查询数据库确认无冲突，有冲突返回 409。

### 3. 管理端使用 antd-img-crop 实现裁剪

`antd-img-crop` 与 antd Upload 组件无缝配合，支持设置 `aspectRatio: 1` 实现 1:1 裁剪。裁剪后自动上传到现有上传接口。

### 4. 移动端头像修改放在个人页

移动端个人页（profile）已有入口，在此处增加头像点击修改能力。裁剪使用移动端图片选择器 + 上传。

## Risks / Trade-offs

- [Risk] 用户名唯一性校验可能在并发场景下冲突。→ 数据库层已有 `unique` 约束，API 层先查再改作为主要防线，极端情况由数据库约束兜底。
- [Risk] 头像文件累积占用磁盘空间。→ 当前规模可控，后续可加清理策略。
- [Risk] antd-img-crop 增加包体积。→ 仅 admin-web 使用，影响有限。
